2. **Falsifier** — predictive: "the symptom only manifests visibly on prototypes whose content is at the frame top (group-bracket) and was hidden on motion-blur where action is centered." That's a *testable* prediction that distinguishes the bug's presence from its absence.
3. **Structure** — concrete: replace CSS scale with inline JS transform; specific function call shape.

The grader hit STRONG (3/3) because all three dimensions were explicitly committed in writing before the fix landed. **This is the reference shape for future commits.**

## Why R2 graded MEDIUM (and how to elevate to STRONG next time)

R2's commit message had:

- **Mechanism** ✓ — `resolveMotionBlur` normalizes 4 input forms; `applyMotionBlur` applied to PLACE branches.
- **Structure** ✓ — concrete helper functions added; ready for replication onto MOVE / FRAME / HANDOFF.
- **Falsifier** ✗ — *no pre-committed falsifier predictions.* The "regression test pair covers all 4 schema forms" line is a unit test (existence test), not a falsifier (bug-shape test).

The distinction matters: a regression test asks *"does this run without error?"* A falsifier asks *"under what conditions would this be wrong?"*

### Examples of pre-committable falsifiers for motion_blur

If you'd added these to the commit message, R2 would have graded STRONG:

- **Negative-space test:** "Applying motion_blur to a static (non-moving) element produces no visual change. If a stationary element shows blur, the axis is wrong."
- **Direction coupling:** "On a `with-from` transit, blur magnitude should scale with transit velocity. A 0px transit must produce 0px blur regardless of `motion_blur: true`."
- **Termination invariant:** "When transit completes, blur clears in ≤ 1 frame. Persistent post-transit blur is a bug."
- **Scope boundary:** "motion_blur applied to a parent does NOT propagate to children with their own motion_blur=false override."
- **Rendering load:** "Motion blur on >N elements simultaneously must not drop the framerate below 30 FPS at 1080×1920. Threshold N is per-platform and worth measuring."

Any 1-3 of these would have flipped the falsifier dimension to PASS.

### Suggested commit-message template for additive features

```markdown
## DESIGN.md item N — <feature name>

**Mechanism:** <how it works, internally — function shape, dispatch logic>

**Structure:** <what's added — helpers, schema fields, dispatch branches>

**Falsifier:** <what would make this wrong>
- <prediction 1>
- <prediction 2>
- <prediction 3>

**Regression coverage:** <prototype path + scenes covered>
```

The existing message style for fixes (R1 shape) already has this. Extending it to additive features would close the gap.

## Net read for the reel-video.skill team

The skill is being developed with **good documentation discipline already**, especially on fixes. The R1 STRONG grade is meaningful — it means the cross-cutting frame-fit fix met the highest standard the 3-leg loop currently checks.

**Single actionable suggestion:** for future additive feature commits (motion_blur-shape items, where the change is a new capability rather than a fix), add a `**Falsifier:**` block listing 1-3 predictions about where/when the feature *would be wrong*. Negative-space tests, scope boundaries, invariants. That single addition would lift these commits from MEDIUM to STRONG in the grader, and — more importantly — gives future-you a way to verify the feature didn't drift away from intended behaviour as the codebase grows.

The grader's job isn't to demand more documentation for its own sake. It's to make the *invariants* explicit so they survive churn. Falsifiers are the cheapest form of that survival.
